1. Overview

This guide establishes the standard Git branching model, pull request workflow, and CI/CD automation for all application repositories.

1.1 Environment Branches

Branch Environment Purpose
dev Development Integration and feature testing
test QA/Staging Quality assurance and UAT
prod Production Live production environment

1.2 Core Principles

Essential Rules
  • Never push directly to dev, test, or prod branches
  • All changes must flow through feature branches and pull requests
  • CI/CD automation handles validation and deployment
  • Code review is required before merging

2. Branch Strategy

2.1 Branch Naming Convention

feature/<feature-name>       # New features
bugfix/<bug-description>     # Bug fixes
hotfix/<critical-fix>        # Urgent production fixes
chore/<task-description>     # Maintenance tasks

Examples:

  • feature/user-authentication
  • bugfix/login-validation-error
  • hotfix/payment-gateway-timeout
  • chore/update-dependencies

2.2 Branch Lifecycle

dev → feature branch → PR → merge → dev
test → hotfix branch → PR → merge → test
prod → hotfix branch → PR → merge → prod

3. Feature Branch Development

3.1 Creating a Feature Branch (CLI)

From dev (most common):

git checkout dev
git pull origin dev
git checkout -b feature/user-profile-page

From test:

git checkout test
git pull origin test
git checkout -b bugfix/api-timeout

From prod (hotfixes only):

git checkout prod
git pull origin prod
git checkout -b hotfix/security-patch

3.2 Creating a Feature Branch (GitHub UI)

  1. Navigate to your repository on GitHub
  2. Click the branch dropdown (usually shows "main" or current branch)
  3. Select the base branch (dev, test, or prod)
  4. Type your new branch name in the search box
  5. Click "Create branch: <branch-name> from <base-branch>"

3.3 Making Changes

# Check current status
git status

# Stage all changes
git add .

# Or stage specific files
git add src/components/Login.js

# Commit with descriptive message
git commit -m "Add email validation to login form"

# Push to remote
git push origin feature/user-profile-page

3.4 Keeping Your Branch Updated

Regularly sync with the base branch:

git checkout feature/user-profile-page
git pull origin dev
If conflicts occur: 
# Resolve conflicts in your editor
git add .
git commit -m "Resolve merge conflicts with dev"
git push origin feature/user-profile-page

4. Pull Request Workflow

4.1 Creating a Pull Request

  1. Navigate to your repository on GitHub
  2. Click Pull requests tab
  3. Click New pull request
  4. Configure the PR:
    • Base branch: dev, test, or prod (where you want to merge)
    • Compare branch: Your feature branch
  5. Add a clear title (e.g., "Add user authentication system")
  6. Write a detailed description
  7. Assign reviewers (if required)
  8. Add labels (e.g., enhancement, bug, urgent)
  9. Click Create pull request

Example PR Description:

## What does this PR do?
Adds email/password authentication with JWT tokens

## Changes made
- Created Login component
- Added auth middleware
- Implemented token refresh logic

## Testing done
- Unit tests for auth functions
- Manual testing on dev environment

## Related issues
Closes #123

4.2 Pull Request Review Process

For Authors

  • Respond to review comments promptly
  • Make requested changes in new commits
  • Mark conversations as resolved when addressed
  • Request re-review after updates

For Reviewers

  • Review code for quality, standards, and security
  • Test functionality if possible
  • Provide constructive feedback
  • Approve when satisfied

4.3 Merging the Pull Request

  1. Ensure all checks pass (CI, tests, code quality)
  2. Obtain required approvals
  3. Resolve any merge conflicts
  4. Click Merge pull request
  5. Choose merge strategy:
    • Merge commit: Preserves full history (recommended)
    • Squash and merge: Combines all commits into one
    • Rebase and merge: Linear history without merge commit
  6. Delete the branch after successful merge

5. CI/CD Pipeline

5.1 CI (Continuous Integration)

Purpose: Validate code quality before deployment

What CI Does:
  • Installs project dependencies
  • Runs linters and code quality checks
  • Executes unit and integration tests
  • Builds the application
  • Runs security scans
  • Fails the pipeline if any step fails

5.2 CD (Continuous Deployment)

Purpose: Deploy validated code to the target environment

What CD Does:
  • Creates a backup of the current deployment
  • Pulls the latest code from the branch
  • Installs/updates dependencies
  • Runs build commands
  • Manages PM2 process lifecycle
  • Verifies deployment health
  • Rolls back on failure (if configured)

5.3 Automatic CICD Execution

Merge Target CI Execution CD Deployment
dev ✅ Runs automatically Deploys to DEV
test ✅ Runs automatically Deploys to TEST
prod ✅ Runs automatically Deploys to PROD
Key Points:
  • CICD triggers only on successful PR merge
  • CI always runs before CD
  • CD only executes if CI succeeds
  • No manual intervention required for automatic deployments

5.4 Manual CI Workflow Execution

Use this to test CI without merging:

  1. Go to your repository on GitHub
  2. Click Actions tab
  3. Select CI workflow from the left sidebar
  4. Click Run workflow (top right)
  5. Select the branch to run CI against
  6. Click Run workflow
  7. Monitor the workflow progress in real-time

When to use manual CI:

  • Testing CI configuration changes
  • Validating feature branch before creating PR
  • Debugging CI failures

5.5 Manual CD Workflow Execution

Use this for controlled deployments:

  1. Go to your repository on GitHub
  2. Click Actions tab
  3. Select CD workflow from the left sidebar
  4. Click Run workflow
  5. Select the target environment:
    • dev → Development server
    • test → QA/Staging server
    • prod → Production server
  6. Click Run workflow
  7. Monitor deployment progress

When to use manual CD:

  • Deploying a specific commit outside normal flow
  • Re-deploying after infrastructure changes
  • Emergency rollback scenarios
  • Testing CD configuration

6. Troubleshooting

6.1 Common Issues

Merge Conflicts

# Update your branch with latest base branch
git checkout feature/my-feature
git pull origin dev

# If conflicts occur, resolve them manually
# Then stage and commit
git add .
git commit -m "Resolve merge conflicts"
git push origin feature/my-feature

CI Failing

  1. Check the Actions tab for error logs
  2. Run tests locally: npm test or yarn test
  3. Fix issues and push again
  4. CI will automatically re-run on new push

CD Failing

  1. Review CD workflow logs in Actions
  2. Check server connectivity and permissions
  3. Verify environment variables are configured
  4. Check PM2 application status on server
  5. Contact DevOps if infrastructure issues

Branch Out of Sync

# Fetch all remote branches
git fetch origin

# Update your local branch
git checkout dev
git pull origin dev
Accidentally Pushed to Protected Branch
  1. Contact repository administrator immediately
  2. Do not attempt to force push
  3. A revert commit may be required

6.2 Recovery Commands

Undo last commit (before push):

git reset HEAD~1

Undo last commit (after push):

git revert HEAD
git push origin <branch>

Discard local changes:

git checkout -- <file>
# Or discard all changes
git reset --hard HEAD

7. Quick Reference

7.1 Essential Git Commands

# Check current status
git status

# View commit history
git log --oneline

# Switch branches
git checkout dev
git checkout test
git checkout prod

# Pull latest changes
git pull origin <branch>

# Stage changes
git add .                    # All files
git add <specific-file>      # Specific file

# Commit changes
git commit -m "Your message"

# Push changes
git push origin <branch>

# View all branches
git branch -a

# Delete local branch
git branch -d feature/old-feature

# View remote URL
git remote -v

7.2 Daily Workflow Checklist

Starting work:
  • ☐ Pull latest code: git pull origin dev
  • ☐ Create feature branch: git checkout -b feature/my-feature
During development:
  • ☐ Commit frequently with clear messages
  • ☐ Keep branch synced: git pull origin dev
  • ☐ Run tests locally
Before creating PR:
  • ☐ All tests passing locally
  • ☐ Code follows project standards
  • ☐ Branch is up to date with base branch
  • ☐ Commit messages are clear and descriptive
After PR approval:
  • ☐ Merge the PR
  • ☐ Delete the feature branch
  • ☐ Verify automatic deployment succeeded
  • ☐ Test in target environment

8. Best Practices

8.1 Branching Best Practices

Do

  • Always create branches from latest dev/test/prod
  • Use descriptive branch names
  • Keep branches short-lived (< 3-5 days)
  • Delete branches after merging
  • Pull before creating new branches

Don't

  • Push directly to dev, test, or prod
  • Create branches from other feature branches
  • Let branches become stale (>1 week)
  • Reuse old branch names

8.2 Commit Best Practices

Do

  • Write clear, descriptive commit messages
  • Commit logical units of work
  • Use present tense ("Add feature" not "Added feature")
  • Reference issue numbers when applicable
  • Commit frequently

Don't

  • Commit commented-out code
  • Include sensitive data (keys, passwords)
  • Make massive commits with unrelated changes
  • Use vague messages like "fixed stuff" or "WIP"

Good commit messages:

Add user authentication with JWT
Fix null pointer exception in payment handler
Update dependencies to latest stable versions
Refactor database connection logic

Bad commit messages:

fixed stuff
WIP
asdfasdf
final version

8.3 Pull Request Best Practices

Do

  • Write detailed PR descriptions
  • Link related issues
  • Keep PRs focused and small
  • Request reviews from appropriate team members
  • Respond to feedback promptly
  • Test your changes before creating PR

Don't

  • Create PRs with failing tests
  • Merge without required approvals
  • Ignore review feedback
  • Create PRs with hundreds of file changes

8.4 CI/CD Best Practices

Do

  • Monitor CI/CD workflows regularly
  • Fix failing builds immediately
  • Test CI changes in feature branches
  • Use manual CD sparingly
  • Review deployment logs after each release

Don't

  • Ignore failing CI checks
  • Bypass CI/CD process
  • Deploy without CI validation
  • Skip testing in lower environments

8.5 Security Best Practices

Do

  • Use environment variables for secrets
  • Rotate credentials regularly
  • Review dependencies for vulnerabilities
  • Enable branch protection rules
  • Require code reviews

Don't

  • Commit API keys, passwords, or tokens
  • Disable security checks
  • Grant unnecessary permissions
  • Ignore security warnings

8.6 Workflow Diagrams

Feature Development Flow

dev branch ↓ Create feature branch ↓ Make changes locally ↓ Commit & push ↓ Create Pull Request ↓ Code review ↓ CI runs automatically ↓ Merge to dev ↓ CD deploys to DEV automatically

Hotfix Flow

prod branch ↓ Create hotfix branch ↓ Fix critical issue ↓ Create PR to prod ↓ Emergency review ↓ CI validates ↓ Merge to prod ↓ CD deploys to PROD ↓ Backport to test and dev
